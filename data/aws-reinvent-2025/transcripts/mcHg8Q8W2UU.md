The presentation identifies three fundamental observability building blocks—metrics, logs, and traces—and explores how multi-cloud deployments amplify traditional observability challenges including increased complexity from multiple tools and service dependencies, process overhead from instrumenting multiple cloud services, and scale issues related to managing multiple telemetry streams, ingestion latency, query performance, and data egress costs. AWS defines multi-cloud adoption as driven by differentiated capabilities, mergers and acquisitions, or regulatory compliance requirements, with AWS committed to supporting open standards-based approaches that provide customers freedom of choice and innovation flexibility. The session presents two primary architectural patterns for multi-cloud observability: the centralized collection pattern and the federated query pattern, with hybrid combinations also possible. The centralized collection pattern consolidates all telemetry data from multiple cloud providers into a single storage location, using collector agents on compute workloads to gather signals, an ingestion layer for buffering and routing, a normalization layer to convert data to common formats, and unified storage and visualization layers for global application views. Open source implementation options include OpenTelemetry Collector for data collection, Kong API Gateway and Kafka for ingestion and buffering, OpenSearch Data Prepper for normalization, OpenSearch for logs and traces indexing, Prometheus for metrics time series, and Grafana for visualization. Managed open source services simplify this architecture by eliminating the need for separate ingestion layers, with Amazon Managed Service for Prometheus supporting up to 1 billion time series, Amazon OpenSearch, and Amazon Managed Grafana providing out-of-the-box security and scalability while maintaining open source interoperability. Cloud-native implementations can leverage Amazon CloudWatch as centralized storage, using the CloudWatch agent for both AWS and Google Cloud workloads, with Google Pub/Sub and AWS Lambda handling infrastructure telemetry export and normalization from Google Cloud into CloudWatch for unified dashboards, troubleshooting, anomaly detection, and automated remediation. The federated query pattern addresses scenarios where data must remain in its original location due to compliance requirements or cost considerations, allowing queries across multiple sources without data movement through a federation layer that splits queries into sub-queries for each cloud provider, aggregates results, and uses metadata catalogs to store schema information for query planning. Open source federated implementations use Trino as query coordinator with Hive Metastore as metadata catalog, executing queries via Trino connectors against data stored in Amazon S3 for AWS telemetry and Google Cloud Storage for GCP telemetry, with Grafana providing visualization. Cloud-native federated approaches utilize Amazon Athena as query coordinator, AWS Glue as metadata catalog, and Amazon Managed Grafana for visualization, processing queries that retrieve data from both cloud providers and aggregating results for unified presentation. Key decision factors between patterns include centralized collection offering single source of truth, faster analytics, better governance, optimized query speed and simplicity, but with increased storage costs, while federated query provides compliance and data residency benefits, data freshness, optimized storage costs, and flexibility, but introduces query latency and more complex orchestration. The presentation includes a customer case study from Phillips 66, a global energy company managing thousands of applications across multiple cloud providers with 70% workload in AWS, who implemented centralized collection pattern with OpenTelemetry, Amazon Managed Service for Prometheus, and Amazon Managed Grafana, achieving 30% faster mean time to resolution. The session emphasizes that pattern selection is not binary, with hybrid solutions possible based on specific use cases, and recommends considering compliance requirements, data freshness needs, and cost sensitivity early in the decision-making process to select the most appropriate observability architecture for multi-cloud deployments.
