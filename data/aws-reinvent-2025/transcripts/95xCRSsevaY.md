  This session introduces Crow, an open-source Kubernetes resource orchestration project that addresses the complexity and inefficiency of deploying multi-component applications using traditional infrastructure pipelines. Shani Cazaz and Petro from AWS begin by illustrating a common enterprise scenario where deploying a voting application with 5 components (3 microservices, a database, and a cache) using Kubernetes for microservices but separate pipelines for database and cache can take a full business day, requiring coordination across networking teams for VPC and subnet provisioning, infrastructure teams for database deployment, platform teams for EKS setup, and potentially compliance pipelines that may delete non-compliant resources without self-healing capabilities, highlighting three fundamental problems with traditional pipelines: they are not composable requiring manual connection between components, they lack self-healing when resources are deleted by compliance checks, and they maintain unclear state management across multiple execution files and infrastructure-as-code definitions. The presenters propose using APIs and Kubernetes controllers as a solution, explaining how Kubernetes' declarative model and reconciliation pattern naturally support infrastructure management, with AWS Controllers for Kubernetes (ACK) enabling deployment of cloud resources through Kubernetes APIs, but noting that at enterprise scale with multi-region, multi-environment deployments, hundreds of clusters, and diverse resource types including third-party integrations, managing numerous individual controllers becomes complex and doesn't provide a central view or composable abstractions. Crow emerges as the solution, described as "all-purpose glue for Kubernetes resources" that was open-sourced by AWS in November 2024 at KubeCon, with Google and Microsoft joining as steering committee members within two months, making it the first open-source project where all three major cloud providers collaborated from the beginning, and in September 2025 Crow moved to Kubernetes SIG, positioning it to become a core Kubernetes feature rather than just another open-source project. Crow's three core capabilities include a human-friendly schema definition system that automatically generates OpenAPI3 schemas under the hood but presents a much simpler interface for end users compared to raw OpenAPI3, CEL (Common Expression Language) integration for defining logical operations, dependencies, and value passing between resources with advanced features like write-time cost budgeting and type checking, and automatic construction of directed acyclic graphs that determine optimal resource provisioning order without manual dependency management. The architecture works by installing the Crow controller in a Kubernetes cluster, which registers a Custom Resource Definition called Resource Graph Definition (RGD), allowing platform, security, and database teams to collaboratively create composable custom APIs that developers can instantiate, with Crow using standard Kubernetes reconciliation patterns to observe changes, compare current state to desired state, and automatically create, update, or delete resources as needed, providing the self-healing capabilities that traditional pipelines lack. A Resource Graph Definition consists of three main parts following Kubernetes conventions: metadata, spec (containing schema definitions for user-configurable fields with default values and validation rules, and status fields that Crow populates for passing values between resources), and a resources section specifying Kubernetes resources, cloud resources, or third-party resources with variables and dependencies. The practical benefit is dramatic simplification for developers, transforming complex multi-step pipeline deployments requiring coordination across multiple teams into simple API calls where developers just provide a name and configuration flags (like enabling or disabling load balancers or cache), with all underlying infrastructure components automatically provisioned, dependencies resolved, and values passed between resources, while maintaining the same architectural structure. The session demonstrates this with a voting application example, showing how developers can enable or disable features through simple boolean flags while Crow handles all the complexity of provisioning databases, caches, networking, and microservices in the correct order with proper dependency resolution, making infrastructure deployment as simple as declaring desired state rather than managing complex pipelines, state files, and manual coordination between teams.
