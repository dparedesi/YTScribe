summary: This session features Joseph Izorri (Director of Product Management at AWS) and Brandon from Scopey, detailing Scopey's journey scaling Monopoly Go, a massively popular mobile game with over a million concurrent users, on AWS using DynamoDB and ElastiCache Serverless (Valy). Brandon explains their architectural choices, emphasizing a C#-based monolithic application with DynamoDB for user-scaling data, MySQL for employee-scaling configuration data, and Redis (now ElastiCache Serverless) for leaderboards and temporary states. The presentation covers three critical aspects: reliability at scale, cost management, and operational simplicity. Regarding reliability, Brandon highlights why DynamoDB is preferred over MySQL for their gaming workload, citing MySQL's single-writer bottleneck, connection limit issues when scaling to 2000+ hosts with millions of concurrent users, vulnerability to unconstrained queries that can bring down the system, and the painful requirement for cluster upgrades that often require downtime, whereas DynamoDB eliminates these concerns entirely. Monopoly Go faces inherent sudden load spikes due to time-based mini-game events that triple baseline traffic within 5 minutes, with certain DynamoDB tables experiencing jumps from zero to 100,000 read or write units per second. To handle these spikes, Scopey leverages DynamoDB on-demand mode with pre-warming APIs to ensure sufficient underlying capacity before events, sets minimum ECPU thresholds for ElastiCache Serverless, configures EC2 auto-scaling group minimums, and sets load balancer capacity reservations, eventually automating all these activities into a unified system. Their DynamoDB strategy involves launching new tables in on-demand mode, calling pre-warming APIs to prepare partitions and capacity, monitoring traffic patterns, and potentially switching to provisioned mode for cost savings if traffic stabilizes, with the flexibility to switch back to on-demand up to four times daily when sudden spikes occur. For throttling mitigation, they implement client-side retries with exponential backoff, with servers giving up quickly but hinting to clients about intermittent failures, allowing graceful degradation rather than crashes. Cost optimization focuses heavily on reducing DynamoDB item sizes, as read and write unit costs are proportional to document size, with Scopey building custom instrumentation to monitor average, P95, and maximum document sizes and setting alarms when documents approach the 400-kilobyte hard limit. A detailed case study demonstrates redesigning a friend invitation table from one large document per user containing a JSON blob of all invites to one document per invite with a local secondary index on date, reducing write units from 1-200 to a fixed 2 and read units from 1-50 to 1-2, with additional savings from pagination allowing users to query only the most recent invitations. The operational simplicity story details their migration from a self-managed Redis cluster that was difficult to scale without disruption, required constant memory monitoring and alarms, and led to Saturday night pages when approaching capacity limits, to ElastiCache Serverless using Valy (20% cheaper than Redis) for auto-scaling, eliminating memory alarms, and achieving an unexpected 61% cost reduction primarily because they no longer needed to over-provision capacity out of fear of causing user disruptions during scaling operations. Despite early adoption challenges with minor 1-3 minute outages that AWS resolved, Scopey achieved their goal of operational simplicity, freeing engineering time from database maintenance and allowing focus on game building rather than infrastructure management.
