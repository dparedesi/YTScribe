summary: This presentation introduces CloudWatch enhancements designed to unify security and observability data within AWS, addressing the critical challenge of data fragmentation where the same log sources are duplicated across multiple specialized tools serving security teams, observability teams, and audit compliance teams, creating operational overhead, cost inefficiencies, and delayed incident response when problems span multiple domains requiring manual communication between teams that cannot see the same landscape or understand problems from shared context. The fundamental problem stems from specialized teams using purpose-built tools with underlying data tied to those specific tools, creating silos where security teams see firewall denials while application teams see traffic spikes without recognizing a DDoS attack, or where the same AWS service logs, CloudTrail logs, application logs, and third-party logs from CrowdStrike and Okta are being plumbed into different places through complex ETL pipeline spider webs that become nightmares to manage, especially when onboarding new workloads or regions. The solution reimagines CloudWatch as a unified data store blurring lines in the underlying data layer rather than forcing teams to use single tools, enabling creation of comprehensive consistent data stores with all log and telemetry data while maintaining flexibility for different personas to use their preferred tools, structured around four critical phases: collection bringing logs from AWS services, applications, and third parties into a single location with consistent arrival, curation shaping data for analytics needs including field identification, dashboard planning, query optimization, log enrichment, transformations, and index creation for fast queries, storage providing flexible retention policies and cross-account cross-region centralization capabilities, and analytics delivering powerful insights through real-time log viewing and deep dive capabilities. Storage innovations include cross-account cross-region centralization bringing log telemetry into single locations through configurable rules, separate retention policies for different data versions, transformation capabilities during centralization allowing different enrichment for centralized versus source stores, and support for both standard class optimized for real-time operational troubleshooting and infrequent access class at half the cost for compliance use cases.
