Rule two emphasizes setting observability from day one using OpenTelemetry-compatible traces, trace-level debugging, and dashboards that serve different personas from end users to developers to platform teams. Rule three focuses on tool exposure, recommending clear descriptions, explicit parameter specifications, error handling guidelines, retry logic, and reuse of MCP servers both external and internal. Rule four covers evaluation strategies, emphasizing automated testing with diverse examples, technical metrics like latency and accuracy, and business metrics that build user trust, with continuous real-time scoring of agent interactions. Rule five addresses multi-agent architectures, explaining when to split agents into specialized components (like data retrieval, analysis, and reporting agents) to improve accuracy, speed, and maintainability, while distinguishing between protocols like A2A and collaboration patterns like sequential, hierarchical, or peer-to-peer. Rule seven advocates using code whenever possible for deterministic operations like calculations, validations, and rule-based logic, reserving agents for reasoning tasks and measuring cost versus value.
