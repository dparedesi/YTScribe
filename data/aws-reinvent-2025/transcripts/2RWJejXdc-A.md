summary: "This session, \"Scaling Multi-Tenant SaaS Delivery with Amazon CloudFront (NET316),\" introduces CloudFront SaaS Manager, a new feature designed to solve the complexity of serving thousands of distinct tenant domains from a single platform, addressing the challenge where SaaS platforms landing global enterprise customers need to support thousands of domains overnight with strict security and uptime demands. Sagar Desara from AWS opens by explaining that multi-tenancy at the edge is fundamentally harder because certificate handling, routing isolation, security boundaries, and per-tenant policy enforcement were traditionally not primitive CloudFront constructs, with traditional approaches failing: approach one using dedicated distributions per tenant doesn't scale because CloudFront distributions aren't meant to mutate thousands of times daily, with each update taking minutes to propagate globally, slowing deployment pipelines; approach two using shared distributions faces the opposite problem of achieving logical tenant isolation while sharing one distribution. The session outlines three tenant isolation strategies: siloed isolation with dedicated distributions offering maximum isolation and security but at higher cost and management overhead, typically reserved for high-value premium clients; pooled isolation where tenants share resources with logical isolation through namespaces, tenant IDs, and per-tenant quotas, highly efficient and scalable but with potential noisy neighbor impacts; and tier-based isolation combining both approaches where premium customers get siloed resources while others are pooled, optimizing for cost and performance without sacrificing security. CloudFront SaaS Manager addresses these challenges through core constructs: multi-tenant distribution as a single CloudFront distribution configured to serve multiple tenants simultaneously with tenant-specific behavior through routing and caching logic, acting as baseline config for the entire SaaS platform; distribution tenants as individual customers or logical entities where every tenant's request, cache entries, and experience are logically isolated even though multiple tenants share the same distribution; tenant parameters as tenant-specific key-value pairs that customize routing and origin behaviors dynamically without touching underlying code, allowing developers to define parameters at the multi-tenant distribution level and assign specific values at the tenant level; cache policies that define what parts of requests (headers, query strings, cookies) are included in the cache key, enabling separate cache entries for each tenant by including tenant identifiers in the cache key; and connection groups as logical partitions controlling how end-user connections are managed, allowing connection pools to be isolated for groups of tenants or per tenant, enabling features like static IP lists for customers requiring firewall allow-listing or Apex domain support. A major innovation is HTTP-based domain validation for SSL certificates, solving the challenge where SaaS providers control few domains but the majority are controlled by end customers, making standard DNS validation impossible; CloudFront integrates automatically with ACM to provision certificates end-to-end without involving end customers, with HTTP validation allowing full automation even for domains where the provider doesn't control DNS, keeping the entire tenant onboarding flow fully automated. Bhagirat demonstrates the console workflow: creating a multi-tenant distribution with wildcard certificates inherited by all tenants, defining parameters for dynamic origin routing (e.g., S3 bucket paths per tenant), associating WAF rules at distribution level inherited by tenants, creating tenants with subdomains covered by wildcard certificates and custom domains using HTTP validation, assigning parameter values per tenant, overriding WAF rules for enterprise customers with custom requirements, and configuring connection groups with static IP lists for Apex domain support. Ryan from Netlify shares their real-world implementation serving over 9 million developers and 6 million websites daily, explaining how they built their own global network initially because existing providers couldn't handle the cardinality problem, then integrated CloudFront SaaS Manager for cost benefits (CloudFront bandwidth dramatically cheaper than Direct EC2), improved DDoS resilience by pushing bad traffic filtering earlier in the network, and leveraging CloudFront's strengths to improve their product. Netlify's automation workflow: customer provides domain information, service translates customer context into AWS API calls, creates distribution tenant with parameters, requests managed certificate via getManagedCertificate API receiving validation redirects that Netlify handles with static redirects (bread and butter for their platform), polls certificate status until issued, associates certificate with tenant via updateDistributionTenant API, and completes onboarding in under 10 seconds, scaling to thousands of tenants with soft limit around 10,000 (can be raised with AWS team).
