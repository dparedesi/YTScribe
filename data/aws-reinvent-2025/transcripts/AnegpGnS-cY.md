summary: Saurabh Kumar and Haresh Nandwani, both senior solutions architects at AWS and members of a technical community focused on resilience and chaos engineering, present a comprehensive session on implementing chaos engineering for serverless workloads using AWS Fault Injection Service (FIS) Native Fault Actions specifically designed for Lambda functions. The presentation addresses a critical gap in resilience testing, as while most attendees use Lambda functions in production, far fewer perform systematic resilience testing on their serverless architectures. The speakers explain that although Lambda is a resilient service by design with multi-AZ redundancy and fault isolation, Lambda functions typically integrate with upstream and downstream systems, making it essential to validate how the overall architecture responds when these functions encounter failures. FIS provides three key Lambda actions: add start delay (to test the impact of Lambda function initialization delays on consuming components), modify integration response (to simulate incorrect responses from Lambda or downstream systems), and invocation errors (to test the impact of failed Lambda invocations on calling components). The implementation leverages Lambda extensions, which are custom code packages that run as separate processes within the Lambda execution environment, using an API proxy pattern to hook into the function's execution request-response lifecycle and inject faults accordingly. The FIS extension implements a dual polling mechanism with slow poll mode (60 seconds by default when no experiments are running) and hot poll mode (20 seconds during active experiments), representing a trade-off between performance overhead and quick recovery to normal state. The speakers demonstrate the complete implementation using CDK code in Java, showing how to configure the necessary S3 buckets for experiment configuration and report storage, set up IAM policies and roles for FIS permissions, create experiment templates with specific parameters like fault duration and invocation percentage, target Lambda functions using resource tags, and integrate CloudWatch dashboards for comprehensive observability. They emphasize that observability is absolutely critical for chaos engineering success, as teams need to clearly identify metrics to track the impact of experiments, including pre-experiment steady state, experiment state, and post-experiment recovery state. The session includes live coding demonstrations deploying the infrastructure, running experiments, and analyzing the generated reports that show the exact timeline of fault injection and system behavior. Additionally, the speakers introduce two new FIS scenarios for testing partial disruptions or "gray failures": AZ application slowdown (adding latency within a single AZ) and cross-AZ traffic slowdown (simulating packet loss between availability zones), enabling more nuanced resilience testing beyond absolute failure scenarios.
