  Luis Girigay outlines how to build reliable, efficient systems by distinguishing resiliency (recover after a crash), reliability (prevent failures), and efficiency (meeting SLOs with minimal resources), then explains that disciplined performance testing underpins reliability and efficiency. Using a photo of himself squatting under a car as an analogy for an app under load, he defines performance testing as learning how behavior changes when weight increases, lasts longer, or shifts suddenly. He walks through test types beyond basic load testing for expected traffic: stress tests to find degradation or 2x–10x limits; endurance/soak tests to expose memory leaks that surface after hours; scalability tests to ensure smooth growth from zero to target load; spike tests to survive instant jumps (e.g., promo at 9 a.m.) and equally rapid drop-offs, ensuring auto scaling can scale up and down fast; and volume tests for massive data ingest/read on databases or storage. Regular testing is essential—before launches, after infra changes (e.g., moving to Graviton), and daily to spot 5–10% latency drifts or unexpected speedups that hint at misbehavior—and should measure p95/p99 latency as the true customer view, TPS/TPS, bandwidth, regional latency (not just same-garage tests), errors, and resource metrics. Performance testing delivers confidence to scale, better user experiences (<3s responses), and cost optimization by finding the “sweet spot” instance size where latency gains plateau instead of overprovisioning; reliability spend reduces later resiliency work. He recommends aligning test scope to real traffic, especially multi-region if customers are global, and repeating tests regularly rather than a single prelaunch run, because many outages are preventable with good testing. Tooling options include JMeter (Java/XML), Grafana k6 (JavaScript), and Locust (Python). For those who don’t want to run their own infra, he demos AWS’s open-source Distributed Load Testing on AWS (v4.0.1): a single-tenant, CloudFormation-deployed solution ready in ~5 minutes with a web UI and APIs, supporting single HTTP targets or JMeter/k6/Locust scripts, schedulable one-offs or recurring runs, CI/CD integration, and true geo-distribution. Users pick regions, virtual users, unique IPs/containers, ramp-up profiles (e.g., 0 to 1M in a minute), and hold durations; the service spins up containers/tasks in each region, drives traffic, collects metrics, and tears down resources automatically. During a live “global sports” example, he shows multi-region ramp-up and a previous run where intentional undersized instances and delayed auto scaling produced errors and slow ramp, illustrating how tests reveal mis-tuned capacity and baseline behavior. He also notes spike tests must check the downscale path to avoid cost surprises or lingering capacity, and recommends monitoring daily baselines to catch anomalies quickly, since both humans and machine-to-machine callers experience the effects of slowness. He closes urging attendees to deploy the solution, establish baselines, iterate continuously, and visit the AWS Village demo for deeper guidance.
