LMI fundamentally allows customers to run Lambda functions on EC2 instances of their choice with access to over 400 instance types across general purpose, compute optimized, and memory optimized families supporting Intel, AMD, and ARM Graviton architectures, while AWS handles all operational elements including instance lifecycle management, operating system patching, runtime updates, routing, and auto-scaling according to customer configurations. The setup process involves three key steps: creating a capacity provider that defines instance-level configuration including VPC settings, instance type preferences, and scaling guardrails, creating a function and associating it with the capacity provider to designate it as an LMI function, and publishing a function version which triggers the actual deployment of EC2 instances and execution environments across availability zones. LMI supports familiar Lambda features including OCI containers and zip packaging, latest versions of Java, Python, Node, and .NET runtimes, layers and extensions, function URLs, response streaming, 15-minute timeouts, and durable functions for longer-running multi-step applications, while introducing new capabilities like execution environment memory per VCPU ratios (2:1, 4:1, or 8:1) that influence instance type selection and support for up to 32GB memory per function. The architecture eliminates cold starts entirely by pre-initializing execution environments when functions are published, defaulting to three execution environments across three availability zones that remain warm and ready to handle invocations immediately, with scaling controlled through min and max execution environment settings that allow pre-provisioning for known peaks or scaling down to zero for dev/test workloads to eliminate costs during inactive periods. A fundamental difference from Lambda default is the multi-concurrency model where single execution environments can handle multiple concurrent invokes simultaneously rather than the singly-concurrent model of default Lambda, with language-specific default concurrency limits that can be overridden and intelligent load balancing through an LMI agent that monitors resource pressure and reroutes traffic when execution environments approach saturation, implementing goodput protection to prevent brownouts by throttling requests when all environments are under pressure rather than blindly routing until failure.
