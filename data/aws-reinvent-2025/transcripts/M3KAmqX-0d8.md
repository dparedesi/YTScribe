  In a workshop-style session, AWS presenters explain how to build serverless agentic AI with the open-source Strands agent SDK, Model Context Protocol (MCP), and AWS serverless primitives, framing agentic AI as the “self-driving car” of LLMs that handles multi-step reasoning, tool calls, and decisions rather than single-shot responses. They introduce the agent loop—user prompt triggers an LLM that plans tool usage, executes tools, records conversation state, and recurses as needed—and stress Strands’ Python simplicity: a few lines create an agent with a system prompt and default model (Anthropic Claude 4), or developers can configure any model (Bedrock, OpenAI, Meta) plus hyperparameters. Tools extend agents by wrapping ordinary Python functions with a `@tool` decorator that adds safety, observability, and prevents infinite loops, while built-in utilities like calculator or HTTP requests reduce coding. System prompts must explicitly define the agent’s persona, available tools, expected outputs, and ordering so the LLM plans correctly; examples include a weather-and-commute assistant combining HTTP weather fetch, a custom commute advisory function, and a dice-rolling MCP tool. MCP is positioned as the open standard that converts the combinatorial M×N integration problem between multiple tools and multiple LLMs into manageable M+N by exposing tools via standard schemas; Strands ships with an MCP client (supporting stdio, SSE, or streamable HTTP) so agents can consume remote MCP servers hosted anywhere, including Lambda behind API Gateway, with authentication via tokens or other schemes. They outline serverless best practices: host both the Strands agent and MCP server on Lambda with API Gateway fronts; keep expensive initializations (e.g., MCP connections) outside the handler to reuse across invocations and reduce cold starts; set API Gateway timeouts as needed (up to one minute without special approval); and prefer identity-based, standardized access to external tools instead of hardcoding. Live coding and demos attempt to deploy via AWS SAM templates and a Makefile to create API Gateway and Lambda resources, swap in a Bedrock Nova Pro model, and test tools individually (weather check, commute advisory) and together, including context carryover and tool chaining. The MCP server example is intentionally simple—a dice roll service—but demonstrates secure access, tool discovery, and invocation through the MCP inspector, including error handling when bad tokens cause 403s. Audience Q&A covers agent-to-agent interactions (A2A), using AgentCore for memory, ordering vs parallel tool calls, handling unavailable MCP servers by falling back to local tools, updating system prompts/configurations externally for flexibility, managing per-user session state via DynamoDB if needed, and trade-offs of letting multiple agents share a single MCP server.
