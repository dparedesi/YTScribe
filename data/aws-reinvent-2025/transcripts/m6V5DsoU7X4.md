The session begins by establishing the fundamental challenge facing modern cloud applications where end users are globally distributed, requiring data proximity for low latency, high availability across regions, and seamless business continuity during regional disruptions, making multi-region database architectures essential rather than optional. The speakers introduce the core trade-offs inherent in distributed systems, particularly the tension between performance and consistency, noting that as geographic distance between regions increases from tens to hundreds of milliseconds, architects must decide whether to prioritize predictable low-latency operations or strong cross-region data visibility, and whether to accept potential data loss measured by Recovery Point Objective during regional failures versus achieving near-zero Recovery Time Objective through active-active architectures. DynamoDB Global Tables addresses these challenges with two distinct replication modes, each optimized for different workload requirements. The asynchronous replication mode, which has been available since the original Global Tables launch, provides eventually consistent replication where writes complete locally within a single region before being asynchronously propagated to other regions via unidirectional replicators that pull from DynamoDB Streams, ensuring that applications experience the same predictable single-digit millisecond latencies regardless of the number of replica regions while accepting that strongly consistent reads only reflect local writes and cross-region visibility occurs eventually, typically within sub-second timeframes under normal conditions. Amrit provides detailed explanations of conflict resolution using last-writer-wins semantics based on atomic clocks synchronized across regions, demonstrating through timing diagrams how concurrent writes to the same item in different regions are resolved deterministically by timestamp, and importantly noting that DynamoDB Streams notifications may differ across regions since stale replicated writes are silently dropped without generating stream events, a critical consideration for applications built on streams-based architectures. The session then explores various failure scenarios, showing that during complete regional isolation, the isolated region cannot replicate to others but remains fully operational for local reads and writes, while network partitions between two regions in a three-region setup do not prevent replication since data can route through the third region as an intermediary, though direct replication between the partitioned regions ceases until connectivity restores. The second half introduces Multi-Region Strong Consistency Global Tables, or "Mercy Global Tables," announced at the previous re:Invent and now generally available with two-region plus witness configurations to reduce costs. This mode fundamentally changes the replication architecture by embedding a multi-region journal within DynamoDB itself rather than relying on external stream-based replicators, where each write operation first generates a replication log entry describing the conditional update needed to maintain idempotence, appends this log to the multi-region journal that requires acknowledgment from at least two of three regions for durability, and only then returns success to the application before asynchronously applying the log entry to all replica tables, ensuring that every region sees writes in the identical journal-determined order regardless of which region originated the write. The critical innovation for strongly consistent reads involves inserting a "heartbeat" or "read fence" log entry into the journal and waiting for it to be applied locally, guaranteeing that all prior writes from all regions have been processed before serving the read, though this adds cross-region round-trip latency making strongly consistent reads approximately as slow as writes. The speakers candidly discuss the trade-offs, noting that while Mercy Global Tables eliminate the read-after-write consistency gap and provide better behavior during network partitions where all regions remain available by routing replication logs through healthy paths, they sacrifice per-operation latency, do not currently support transactions or Time-To-Live features, and can experience write conflicts requiring application-level retries when concurrent updates target the same item from different regions, as the conditional updates in replication logs may fail when applied out of order.
