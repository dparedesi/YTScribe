The fundamental difference between purpose-built models (trained on business data, singular purpose, know a lot about your business) and foundation models (trained on public data, infinitely reusable, democratized access, but lack business context) creates the core challenge: during prompt time assembly, enterprises must gather contextual information from thousands of databases, SaaS applications, and APIs scattered across data silos, making data strategy essential for AI strategy. The evolution from single-turn prompts through workflows (chaining multiple prompts with deterministic logic, limited by edge case planning and error correction challenges) to agents (LLM with instructions and tools working on open-ended tasks, able to recover from errors and adjust course) represents a shift in software engineering where business logic becomes an output of data and context rather than precise code, moving from iterating on code to iterating on data with probabilistic testing through evals rather than deterministic unit tests. The shift from prompt engineering (crafting precise instructions when models weren't very smart, focusing on formatting tricks and XML tags) to context engineering (managing the entire information environment including system prompts, user messages, tools, assistant messages, tool calls and results building up over time) addresses three key levers: context window limits (API enforces maximum tokens), context rot (dumping too much text hurts performance even before hitting limits), and prompt caching benefits (fixed prefixes enable cost discounts and latency improvements). System prompt best practices focus on the Goldilocks zone: just the right amount of detail leaving things open-ended for the model to handle nuance, avoiding too prescriptive approaches with complex if-else logic encoding every scenario, and avoiding too vague instructions that confuse the model, with the test being whether a friend could accomplish the task with those instructions. Tool design treats tool definitions as additional prompting (tools array gets plopped at the top of system prompt), requiring simple accurate names, detailed well-formed descriptions with examples of good and bad usage times, and avoiding too similar tool names especially for search tools that can confuse the model about when to use which tool. Data retrieval has evolved from RAG (retrieve-then-generate where bad retrieval means the LLM can't recover) to agentic setups where agents can use search tools, write queries, get results, and try again if they didn't get what they wanted, requiring thinking about what to preload (instructions and tools) versus what to hide behind tools for progressive disclosure. Skills build on this pattern, allowing end users to define markdown or Python files that might be useful at certain times, with the system prompt directing agents to look in skills folders when needed rather than loading everything upfront. Long horizon optimizations address context window filling up through techniques like compaction (detecting when approaching limits, injecting a user message asking for summary, clearing conversation, starting fresh with summary), experimental memory where models write notes for themselves, and sub-agent architectures. The core data retrieval challenge requires marrying fresh real-time data with derived curated data sets, as human reasoning pairs real-time signal information with historical patterns, and operational use cases like patient monitoring or customer service need current state information, not stale reports.
