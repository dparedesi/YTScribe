summary: This comprehensive presentation introduces spec-driven development as a structured approach to AI-powered software development, addressing the common "prompt and pray" problem where developers iterate repeatedly without clear requirements, design decisions, or documentation. Spec-driven development consists of three main artifacts: requirements documents that use industry-standard EARS (Easy Approach to Requirements Syntax) to define user stories, acceptance criteria, and target audiences; design documents that outline technical implementation, architecture decisions with rationale, system diagrams, security considerations, and performance requirements; and task files that break down implementation into discrete, granular prompts for the agent to execute. The workflow emphasizes planning first before writing code, with continuous feedback loops between the developer and agent at each phase, ensuring requirements are ironed out, designs are reviewed, and technical decisions are documented before any implementation begins. The presentation highlights how this approach brings structure to AI coding, moving from prototype to production more reliably by eliminating ambiguity, documenting critical decisions upfront, and breaking work into atomic, independently verifiable chunks. A significant innovation discussed is property-based testing (PBT), introduced in Quiro's GA release, which automatically extracts system properties and invariants from requirements and design documents, generating comprehensive test suites that can verify thousands of input permutations without manual test case creation. These properties are tied back to specific requirements, providing traceability and confidence that implementations meet their original specifications. The presentation includes detailed case studies from the Quiro engineering team, demonstrating how they used spec-driven development to ship features like native notifications in 48 hours, remote MCP support with full design reviews, and dev server support with long-running process management, all while working with unfamiliar codebases or complex integrations. The session emphasizes the importance of providing rich context to agents through steering files that document tech stacks, project structure, and product information, as well as leveraging MCP servers to pull requirements from external systems like Jira, Linear, or Asana. The live demo showcases building an authentication feature for a bike-sharing application, starting with fuzzy requirements from a Jira ticket, using MCP to fetch detailed requirements and AWS Amplify documentation, generating comprehensive specs with refined user stories and acceptance criteria, creating design documents with architecture diagrams and data flow visualizations, and finally implementing tasks that integrate authentication with proper state management, session persistence, and conditional UI rendering. The presentation concludes with best practices: always start simple and avoid overcomplicating problems, use function calling for deterministic outputs, employ single agents for multi-step adaptive reasoning tasks, deploy multi-agent systems for complex domain collaboration, commit specs to version control for team collaboration and historical reference, and continuously refine specs as requirements evolve, treating them as living documents that can be reopened and updated as the system grows.
