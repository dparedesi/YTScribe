The core challenge in production AI workloads is balancing three critical axes: model accuracy, inference speed and latency, and cost, where using smarter models typically means higher prices and increased latency, while accepting "good enough" answers can reduce costs and improve speed. The new tier system addresses the previous limitation where Bedrock only offered on-demand requests without differentiation between high-priority traffic requiring immediate processing and less critical workloads that could tolerate delays. The reserved tier provides dedicated capacity reserved on behalf of customers, guaranteeing availability even during high demand periods when other customers may experience throttling, with flexible provisioning of input and output tokens per minute based on use case requirements, supporting burst to standard tier when reservation capacity is exceeded, and featuring explicit prompt caching with burn-down rate conversions that allow reserving for actual processed tokens rather than cached tokens. The priority tier is a pay-as-you-go option priced at 75-100% premium over standard tier, designed for spiky sporadic traffic that is still important and cannot tolerate multiple retries or throttling, such as check deposit verification in digital banking apps where customers would abandon the process if requests fail, featuring optimized fleet configurations with smaller batch sizes for faster output token processing, higher priority queue placement, and prompt caching with 90% discount applied to the premium rate. The standard tier represents the existing Bedrock on-demand experience for day-to-day workloads that can tolerate some retries, with best-effort service that includes alarming frameworks and capacity rebalancing when throttling exceeds thresholds, supporting explicit prompt caching across all tiers with 90% discount on input token processing. The flex tier is a new offering providing approximately 50% discount for latency-tolerant workloads like agentic automated workflows where no human is waiting for responses, experimental beta testing workloads, and non-critical real-time requests where processing can take minutes instead of seconds, with prompt caching support ensuring cost benefits are maintained. Intuit's experience demonstrates practical application: using reserved capacity for seasonal TurboTax workloads with predictable token profiles, priority and standard tiers for daily spiky interactive traffic where reserved capacity would cause underutilization, and flex tier for experimental non-critical workloads requiring real-time processing but not production-ready status. The key insight is that priority tier's premium pricing, when amortized over a year, is more cost-effective than reserving capacity 24/7 for sporadic high-priority traffic, while reserved capacity makes sense only when usage patterns are predictable and steady enough to justify fixed hourly costs.
