summary: This presentation addresses a critical design decision facing organizations adopting generative AI: selecting the appropriate architectural approach for integrating AI-powered workflows into real-world systems. The session explores three primary paradigms, each offering distinct benefits and trade-offs that impact performance, safety, and business value. The simplest approach is function calling, where modern large language models like those available on Amazon Bedrock interface with external systems through defined APIs and functions. This feature allows LLMs to invoke functions when accessing external data, performing calculations, or executing code operations, replacing unpredictable free-form text generation with structured, deterministic outputs. Function calling serves as a bridge between probabilistic LLM reasoning and deterministic software APIs, making AI safer for real-world business workflows by enabling reliable data retrieval, calculations, inventory checks, customer balance verification, and booking detail confirmation. The presentation illustrates this with a travel planning scenario where function calling enables an LLM to retrieve actual weather forecasts for specific dates and locations, providing accurate and personalized recommendations rather than generic advice. However, function calling is most effective for known and simple interactions; as enterprise workflows scale to include dozens of tools, interdependent functions, and dynamic requirements, the system requires extensive back-and-forth between the LLM and application, necessitating more sophisticated approaches. The next level is the agent architecture, built on the ReAct pattern that demonstrates LLMs can do more than predict text or call functions directly—they can reason through problems step by step. An agent is an autonomous AI process that receives a goal, reasons on how to solve it, decomposes tasks into multiple steps, and dynamically decides when to call functions, use tools, or fetch external data. Unlike function calling where business logic remains in the application, agents integrate reasoning capabilities, allowing them to plan multiple steps ahead, analyze intermediate results, and adjust actions accordingly. The travel scenario is extended to show an agent autonomously handling multiple aspects: retrieving weather forecasts, checking hotel availability, verifying flights, and suggesting activities based on stay length and season, all while reasoning through the problem and deciding when to invoke each tool. When business problems become more complex and require specialized roles across different domains, multi-agent systems become necessary. Instead of a single agent juggling multiple domains, multiple specialized agents collaborate, each focused on a particular area with dedicated expertise, tools, and instructions. The presentation describes two foundational multi-agent architectures: the hierarchical supervisor pattern with a central orchestrator that breaks down goals into subtasks and distributes them to specialized worker agents, and the network swarm pattern where multiple agents operate as peers in a mesh with shared communication and coordination. Multi-agent systems offer benefits like modularity, where improving one agent doesn't risk breaking others, and resilience, but they also introduce significant challenges including orchestration complexity, increased latency and costs from multiple LLM calls, debugging difficulties in distributed systems requiring detailed observability, and security concerns where each agent's access to sensitive data must be carefully controlled. The presentation emphasizes that choosing between function calling, agents, and multi-agent architectures should be driven by task complexity, the need for predictability versus adaptability, cost and latency constraints, control over business logic requirements, and desired autonomy levels. Simple tasks like retrieving product prices benefit from function calling, moderate complexity tasks like vacation planning suit single agents, while complex enterprise scenarios like legal advisory systems require multi-agent orchestration. The session concludes with key recommendations: always start simple and avoid overcomplicating problems, use function calling when deterministic output is sufficient, employ single agents for multi-step tasks needing adaptive reasoning, deploy multi-agents for different domain specialization and complex workflows, and always plan for observability using tools like AWS Bedrock, Lambda, Step Functions, and Agent Core to build agentic AI solutions with proper monitoring, cost control, and governance.
