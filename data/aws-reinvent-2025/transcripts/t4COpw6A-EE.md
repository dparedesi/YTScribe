The session begins by establishing a methodology for performance tuning, focusing on key areas including CPU utilization from suboptimal queries and full table scans, memory issues from process-based architecture and overconfigured parameters, storage and IOPs problems from multi-version concurrency control dead tuples and unused indexes, and application patterns involving blocking queries and idle transactions, with the recommended approach starting from Database Insights or Performance Insights to identify top queries and weight events, then using EXPLAIN ANALYZE with buffers option to investigate execution plans. The second optimization shows creating partial indexes for frequently filtered columns, where a query filtering for active products (only 33% of the table) was improved by creating a partial index on the category column with a WHERE active = true clause, reducing rows removed by filter from 30,000 to 900, execution time from 360 milliseconds to 23 milliseconds, and buffer hits from 13,000 to 10,000 blocks, with the presenter explaining that partial indexes are superior to composite indexes when you always filter for a specific subset of data. The third optimization addresses query plan management using Aurora's APG plan management extension, demonstrating how to capture query plans, identify when hints force suboptimal index usage, and switch between plans without application code changes by approving better plans and rejecting poor ones, showing how a query using a single-column index with 400,000 rows removed by filter and 400,000 buffer hits was switched to use a composite index with zero rows removed and only 37,000 buffer hits.
