The migration strategy employs the strangler fig pattern, gradually breaking out services from the monolith while leaving the rest intact, starting with services like payment, metrics, and billing on EC2 instances, but this creates new challenges with hundreds of EC2 instances having different configurations and deployment pipelines, leading to container adoption for consistent environments, dependency management, and reduced operational complexity, though containers introduce the need for orchestration at scale. Kubernetes emerges as the solution for orchestrating containers, providing service discovery, load balancing, auto-scaling, and rolling deployments, with Amazon EKS offering a managed Kubernetes environment that eliminates control plane management complexity, provides native AWS integration with IAM, security groups, and database services, and enables running Kubernetes anywhere. EKS Hybrid Nodes allow connecting EKS clusters to VMware nodes or on-premise infrastructure in data centers, enabling microservices to run on managed instances in the cloud while maintaining on-premise services for compliance or data availability requirements. As organizations grow and gain customers, the single-tenant architecture where each customer gets their own cluster becomes operationally inefficient despite providing excellent isolation, leading to the need for multi-tenant architectures that efficiently use infrastructure while maintaining security and compliance. The multi-tenant architecture requires namespace-based isolation to reduce cluster count and create logical tenant boundaries, but Kubernetes networking is flat by default, allowing pods from different namespaces to communicate, necessitating network policies that create digital fences limiting network flow between tenants, combined with ingress gateways for path-based routing and AWS CNI enabling ENI creation for each pod with security group enforcement. Resource quotas and limit ranges address the noisy neighbor problem where one tenant's runaway workload could consume all CPU or memory, with resource quotas setting namespace-level limits on CPU, memory, and pod counts, and limit ranges constraining individual pod resource consumption to prevent machine starvation, though this still requires managing EC2 nodes and capacity planning.
