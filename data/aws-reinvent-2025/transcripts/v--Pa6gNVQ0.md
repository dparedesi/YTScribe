summary: This presentation introduces Target Optimizer, a new feature for AWS Application Load Balancer designed to address load balancing challenges for low concurrency applications such as large language models, media generation services, and other compute-intensive workloads where each instance can only process a small number of requests concurrently, sometimes as few as one request at a time. The session begins by explaining traditional load balancing approaches like round robin and least outstanding requests, which work well for high concurrency applications where each target handles thousands of concurrent requests, but fall short for low concurrency applications where targets can only process 1, 5, 10, or 100 requests simultaneously. The problem is illustrated with an example where targets are busy processing requests of varying complexity, and when new requests arrive, traditional algorithms may route them to already busy targets instead of idle ones, leading to decreased success rates, increased error rates, higher perceived latency from retries, and poor target group utilization with idle targets not receiving requests. The presentation explains that even advanced algorithms like least outstanding requests have limitations because Application Load Balancer consists of multiple independent nodes that make routing decisions independently, so each node has a different view of target capacity, potentially causing multiple nodes to route requests to the same busy targets. Target Optimizer solves this by installing an AWS-provided agent on each target that serves as a proxy between the load balancer and the application, establishing control channels with load balancer nodes to exchange management traffic. The agent tracks the number of requests the application is processing and compares it with a configurable maximum concurrent requests setting, sending signals to load balancer nodes when capacity becomes available. When a request completes and the application can handle another request, the agent notifies a node, which registers that signal and knows it can route the next request to that target. This enables the load balancer to know exactly which targets are idle or have available capacity, and nodes coordinate to avoid sending more requests to a target than it can handle. The solution provides two key benefits: very high request success rates because requests only go to targets with available capacity, and very high target group utilization because capacity is immediately filled as soon as it becomes available, allowing organizations to operate with smaller target group sizes. Additionally, Target Optimizer provides automatic load shedding when all targets are at capacity, protecting targets from being overwhelmed or experiencing denial of service. The setup process involves three steps: installing and configuring the agent on each target with HTTP/HTTPS port, destination port, control port, and maximum concurrent requests setting; registering targets with a new optimized target group that specifies the control port; and adding the optimized target group to an Application Load Balancer listener, allowing gradual traffic migration from existing target groups. The feature supports EC2 instances, IP addresses, Amazon EKS, and ECS services, with the agent running as a sidecar container for containerized workloads. New CloudWatch metrics are available for monitoring including active control channels, channel errors, request counts, rejections, and control queue lengths. The presentation emphasizes that Target Optimizer is particularly valuable for AI inference workloads, media processing, and other compute-intensive applications where traditional load balancing algorithms cannot efficiently distribute load across large fleets of low-concurrency targets.
