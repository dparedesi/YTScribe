  This session introduces Amazon Elastic VMware Service (EVS), a solution for lifting and shifting VMware workloads from on-premises environments to AWS while maintaining the same people, processes, and technology, addressing the reality that cloud migration is not one-size-fits-all and different workloads require different migration strategies. Andy and Aarti from AWS begin by using a moving metaphor to illustrate that just as moving a couch differs from moving books or lamps, migrating applications requires varied approaches including lift-and-shift, re-platforming, refactoring, re-hosting, and relocation, with EVS specifically designed for organizations running VMware on-premises who want to move to cloud-based VMware without changing their operational model. EVS runs the standard VMware Cloud Foundation (VCF) 5.2.1 stack on AWS bare metal infrastructure (i4i.metal instances) within customer VPCs, providing the same commercial off-the-shelf VMware software used on-premises, not a modified version, enabling customers to maintain existing skills, processes, third-party solutions, and scripts while gaining cloud benefits like dynamic scaling and access to AWS services. The service operates in a bring-your-own-license model where customers can use existing VCF subscriptions, provides full administrative access (customer-managed, not a managed service), supports both self-management and partner-managed operations for organizations whose managed service providers are no longer part of Broadcom's program, and offers flexible consumption through on-demand, 1-year, or 3-year savings plans. EVS consists of three key building blocks: compute using i4i.metal instances certified by Broadcom for VCF 5.2.1 with a 4-host minimum cluster that can scale up to 16 hosts per environment (with multiple environments connectable via Transit Gateway for larger scale), storage with 30 tebibytes of local NVME per instance using VSAN Express Storage Architecture (ESA) for primary storage with full vCenter control over RAID levels and failure tolerance, plus integration with Amazon FSX for NetApp ONTAP and third-party ISV storage solutions like Pure Storage for additional storage capacity without adding compute hosts, and networking that brings VLAN and subnet concepts into VPC through overlay and underlay networks with AWS Route Server providing BGP propagation between NSX edges and route tables, enabling seamless connectivity between EVS environments, other AWS workloads in the same VPC, and on-premises networks via Transit Gateway with Direct Connect or VPN. The deployment process collects configuration inputs through the AWS console (VPC selection, appliance IP addressing, route server endpoints, DNS hostnames), bootstraps metal instances, pushes configuration to Cloud Builder, and deploys the complete VCF stack in approximately 3 hours, with credentials managed in AWS KMS and full access to SDDC Manager, vCenter, and NSX Manager for day-two operations. Pricing includes EC2 instance costs (with savings plans and reserved instances applicable), VPC Route Server endpoint costs (two per environment), and a service control plane hourly fee for deployment automation and underlying connectivity management. The session features a customer case study from Jignesh Shah at NYU Langone Health, a large academic healthcare system with 53,000 employees serving 2 million daily outpatient visits, who needed to accelerate cloud migration from an average of 320 days per application to weeks while maintaining business continuity critical for healthcare operations. NYU chose EVS because it runs in their own VPC providing full visibility and control (unlike vendor-managed VPC solutions), supports infrastructure-as-code integration with existing CI/CD pipelines, enables bring-your-own-license for existing VCF subscriptions, allows starting with a 4-node pilot light cluster and scaling on-demand, maintains their heterogeneous partner ecosystem (Pure Storage, FSX for NetApp ONTAP), extends NSX security to the cloud for multilayer security, and charges a flat rate per node allowing unlimited VMs per node. Key lessons learned from NYU's deployment include the need to build custom modules for operationalizing EVS integration with on-premises service management systems, challenges with network routing between EVS overlay and VPC underlay requiring time to establish proper policies, unexpected data ingress costs that were resolved with AWS account team assistance, and a learning curve for CloudFormation since NYU wasn't previously a CloudFormation shop. The session emphasizes that EVS is ideal when speed matters for migration, such as when data center contracts are expiring or server support is ending, with the key advantage being the ability to extend Layer 2 networks from on-premises to cloud using VMware HCX technology, allowing VM migration without IP address changes, dramatically reducing testing burden and enabling faster migration with minimal downtime, while customers can gradually modernize by moving databases to RDS while keeping applications in VMware, or fully leverage cloud-native services like S3 and EFS co-resident with VMware workloads in the same VPC.
