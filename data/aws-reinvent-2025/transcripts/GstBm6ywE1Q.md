summary: This session from Anthropic explores long-horizon coding agents capable of building complete applications across multiple sessions, addressing the fundamental challenge that LLMs cannot maintain persistent memory between sessions. The presentation begins by reframing LLM forgetfulness as a feature rather than a bug, since forgetting prevents models from reinforcing bad habits or incorrect assumptions, while external memory solutions enable agents to maintain context across sessions without accumulating errors. The core insight is that agents can return to flow state much faster than humans (who may take 20+ minutes after context switching), but require proper context injection at session start to know what to do next. The human-AI division of labor positions humans as architects responsible for creating vision, requirements, standards, constraints, and architectural decisions, while AI agents act as builders implementing features, writing tests against specified criteria, updating documentation continuously, and iterating on feedback, with a critical constraint that standards files remain read-only to prevent agents from modifying architectural decisions. The environment-as-memory approach uses four key components: a feature list JSON file (preferred over markdown to reduce likelihood of agent editing), a standards.md file ensuring consistent coding style across sessions, a progress.md file tracking where previous sessions left off and integrated with Git commits, and an init.sh file handling deterministic environment setup like starting dev servers. The agentic loop workflow begins each session by examining the environment and previous state, running the test suite to verify previous work remains functional, picking up the next available feature from the specification, implementing the feature while continuously running tests, and committing work with progress notes before the session ends. A critical breakthrough was moving from one-shot implementations (which caused context window exhaustion and premature victory declarations) to incremental progress working on one feature at a time, leaving environments in clean states using commits as memory rather than maintaining persistent state, and implementing end-to-end testing with browser automation to verify features work in practice rather than just in theory. The session demonstrates a live architecture using GitHub issues as a backlog with voting and approval mechanisms, GitHub Actions for CI/CD with an issue puller on cron jobs, AWS Agent Core Runtime for infrastructure orchestration (using Git repositories as state since EFS volumes aren't supported), and Claude Opus 4.5 via the Cloud Agent SDK with enhanced vision capabilities for screenshot analysis. Key learnings include requiring agents to write full end-to-end workflow tests rather than individual screen tests to prevent reward hacking behavior where agents modify tests to pass, using XML-structured specifications for better model parsing than markdown, implementing browser automation with Playwright MCP for visual verification, and having agents commit work even if incomplete when sessions time out to preserve progress. The presentation emphasizes that this approach changes developer workflows from writing code and debugging manually to defining requirements, reviewing AI test results, and verifying outputs, with documentation generated automatically and progress maintained by agents. Limitations include vision gaps where models may miss visual bugs requiring human judgment, browser automation limitations with native alerts, and the need for human review before production deployment since this is not a fire-and-forget solution. The session concludes with core principles: mimic real developer workflows including reading docs and understanding current state before making changes, leave things cleaner than found (writing tests and checking for browser log errors), and test before declaring victory to avoid surprising failure modes that one-shot implementations might produce.
